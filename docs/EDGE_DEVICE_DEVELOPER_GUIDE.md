# Edge Device Developer Guide

This guide is for developers who are building edge devices (e.g., Raspberry Pi) that will send measurement data to the JSEnergy Dashboard system via REST API.

## Quick Reference

**Staging Environment:**
- **API Base URL**: `http://jsenergyapi.aitech.work`
- **Dashboard URL**: `http://jsenergy.aitech.work`

**Development Environment:**
- **API Base URL**: `http://localhost:8000`

## Overview

Edge devices communicate with the JSEnergy Dashboard API to send real-time measurement data. The workflow is:

1. **Device Registration**: An administrator or installation owner registers your device via the web dashboard or API
2. **Token Provisioning**: You receive a unique device token (shown only once)
3. **Authentication**: Use the token to authenticate all API requests
4. **Data Submission**: Send measurement data to the appropriate endpoints (supports batch operations for efficiency)

## Device Registration

### Prerequisites

Before registering a device, you need:
- An **installation ID** - The installation this device will monitor
- Access to the web dashboard or API with a user account that has access to the installation

### Registration Process

Device registration must be done by an administrator or installation owner through the web dashboard or via the API. As a device developer, you typically won't perform this step yourself, but you need to understand the process.

#### Via API (for administrators)

**Endpoint:** `POST /api/installations/{installation_id}/edge-devices`

**Authentication:** Requires user JWT token (Bearer token)

**Request:**
```json
{
  "name": "Raspberry Pi 1",
  "installation_id": 1,
  "description": "Main monitoring device at Building A"
}
```

**Response:**
```json
{
  "id": 1,
  "device_id": 1,
  "installation_id": 1,
  "name": "Raspberry Pi 1",
  "token": "abc123xyz789...",
  "description": "Main monitoring device at Building A",
  "is_active": true,
  "last_seen_at": null,
  "created_at": "2025-01-11T15:00:00Z",
  "updated_at": "2025-01-11T15:00:00Z"
}
```

**⚠️ CRITICAL:** The `token` field is **only returned once** during device creation. You must save this token immediately - it cannot be retrieved later. If lost, the token must be regenerated by an administrator.

### Environment Configuration

After receiving your device token, configure your edge device with these environment variables:

```bash
# Required
DEVICE_TOKEN=abc123xyz789...
API_BASE_URL=http://jsenergyapi.aitech.work
INSTALLATION_ID=1
DEVICE_ID=1

# Optional (for development)
API_BASE_URL=http://localhost:8000
```

**Base URLs:**
- **Staging API**: `http://jsenergyapi.aitech.work`
- **Staging Dashboard**: `http://jsenergy.aitech.work`
- **Development**: `http://localhost:8000`

**Security Notes:**
- Never commit tokens to version control
- Store tokens securely on the edge device (use environment variables or secure storage)
- Rotate tokens if compromised (contact administrator to regenerate)

## Authentication

Edge devices authenticate using a device token, not user JWT tokens. The token must be included in every API request.

### Authentication Methods

You can authenticate using one of two methods:

#### Method 1: X-Device-Token Header (Recommended)

```
X-Device-Token: <device_token>
```

#### Method 2: Authorization Header

```
Authorization: Device <device_token>
```

### Example Request

```bash
curl -X POST "http://jsenergyapi.aitech.work/api/installations/1/measurements/battery" \
  -H "X-Device-Token: abc123xyz789..." \
  -H "Content-Type: application/json" \
  -d '{
    "soc_percentage": 65.5,
    "power_kw": -2.3,
    "voltage": 48.0,
    "temperature": 25.5,
    "timestamp": "2025-01-11T16:00:00Z"
  }'
```

### Authentication Errors

If authentication fails, you'll receive:

**Status Code:** `401 Unauthorized`

**Response:**
```json
{
  "detail": "Invalid or inactive device token"
}
```

Common causes:
- Token is incorrect or missing
- Device has been deactivated
- Device or installation has been deleted

## Configuration Endpoints

Before sending measurements for inverters or EV chargers, you need to know their IDs. Use these endpoints to retrieve the list of devices configured for your installation.

**Base URL:** Use your configured `API_BASE_URL` (e.g., `http://jsenergyapi.aitech.work` for staging or `http://localhost:8000` for development)

**Authentication:** Required on all endpoints (see [Authentication](#authentication))

### List Inverters

**Endpoint:** `GET /api/installations/{installation_id}/inverters`

**Response:**
```json
{
  "inverters": [
    {
      "id": 1,
      "inverter_number": 1
    },
    {
      "id": 2,
      "inverter_number": 2
    }
  ],
  "total": 2
}
```

**Usage:** Use the `id` field from the response as the `inverter_id` when sending inverter measurements.

### List EV Chargers

**Endpoint:** `GET /api/installations/{installation_id}/chargers`

**Note:** This endpoint is only available if the EV chargers feature is enabled for your installation.

**Response:**
```json
{
  "chargers": [
    {
      "id": 1,
      "charger_number": 1
    },
    {
      "id": 2,
      "charger_number": 2

  ],
```

**Usage:** Use the `id` field from the response as the `charger_id` when sending EV charger measurements.

**Example:**
```bash
curl -X GET "http://jsenergyapi.aitech.work/api/installations/1/chargers" \
```

## Data Endpoints

All measurement endpoints follow the pattern:
```
POST /api/installations/{installation_id}/measurements/{component}
```

**Base URL:** Use your configured `API_BASE_URL` (e.g., `http://jsenergyapi.aitech.work` for staging or `http://localhost:8000` for development)

**Authentication:** Required on all endpoints (see [Authentication](#authentication))

**Content-Type:** `application/json`

### Battery Measurements

**Endpoint:** `POST /api/installations/{installation_id}/measurements/battery`

**Request Body:** Array of battery measurements (batch support)
```json
[
  {
    "soc_percentage": 65.5,
    "power_kw": -2.3,
    "voltage": 48.0,
    "temperature": 25.5,
    "timestamp": "2025-01-11T16:00:00Z"
  },
  {
    "soc_percentage": 66.0,
    "power_kw": -2.1,
    "voltage": 48.1,
    "temperature": 25.6,
    "timestamp": "2025-01-11T16:00:05Z"
  }
]
```

**Field Descriptions:**
- `soc_percentage` (required): State of charge percentage (0-100)
- `power_kw` (required): Battery power in kW (negative = charging, positive = discharging)
- `voltage` (optional): Battery voltage in volts
- `temperature` (optional): Battery temperature in Celsius
- `timestamp` (required): ISO 8601 timestamp in UTC (e.g., `2025-01-11T16:00:00Z`)

**Batch Requirements:**
- At least one measurement is required
- Each measurement must have a unique timestamp within the batch
- Duplicate timestamps within a batch will result in `400 Bad Request`

**Response:** `201 Created` with array of created measurement objects

### Inverter Measurements

**Endpoint:** `POST /api/installations/{installation_id}/measurements/inverter/{inverter_id}`

**Request Body:** Array of inverter measurements (batch support)
```json
[
  {
    "power_kw": 3.2,
    "energy_kwh_daily": 12.5,
    "curtailment_percentage": 0.0,
    "timestamp": "2025-01-11T16:00:00Z"
  },
  {
    "power_kw": 3.5,
    "energy_kwh_daily": 12.8,
    "curtailment_percentage": 0.0,
    "timestamp": "2025-01-11T16:00:05Z"
  }
]
```

**Field Descriptions:**
- `power_kw` (required): Inverter power output in kW (positive = generating)
- `energy_kwh_daily` (optional, default: 0.0): Daily energy production in kWh
- `curtailment_percentage` (optional, default: 0.0): Curtailment percentage (0-100)
- `timestamp` (required): ISO 8601 timestamp in UTC

**Note:** The `inverter_id` in the URL path must match an existing inverter for the installation. Use the `GET /api/installations/{installation_id}/inverters` endpoint to retrieve the list of available inverter IDs.

**Batch Requirements:**
- At least one measurement is required
- Each measurement must have a unique timestamp within the batch
- Duplicate timestamps within a batch will result in `400 Bad Request`

**Response:** `201 Created` with array of created measurement objects

**Request Body:** Array of meter measurements (batch support)

```json
    "import_kwh": 12.6,
    "export_kwh": 28.9,
    "l1_a": 2.4,
    "l2_a": 2.2,
    "l3_a": 2.5,
    "timestamp": "2025-01-11T16:00:05Z"
  }
]
```

**Field Descriptions:**
- `import_kw` (required): Grid import power in kW
- `export_kw` (required): Grid export power in kW
- `import_kwh` (optional, default: 0.0): Cumulative import energy in kWh
- `export_kwh` (optional, default: 0.0): Cumulative export energy in kWh
- `l1_a`, `l2_a`, `l3_a` (optional, default: 0.0): Phase currents in amperes
- `timestamp` (required): ISO 8601 timestamp in UTC

**Batch Requirements:**
- At least one measurement is required
- Each measurement must have a unique timestamp within the batch
- Duplicate timestamps within a batch will result in `400 Bad Request`

**Response:** `201 Created` with array of created measurement objects

### Generator Measurements

**Endpoint:** `POST /api/installations/{installation_id}/measurements/generator`

**Note:** This endpoint is only available if the generator feature is enabled for your installation.

**Request Body:** Array of generator measurements (batch support)
```json
[
  {
    "status": "off",
    "fuel_consumption_lph": 0.0,
    "charging_power_kw": 0.0,
    "timestamp": "2025-01-11T16:00:00Z"

  {
    "status": "on",

    "charging_power_kw": 5.0,
    "timestamp": "2025-01-11T16:01:00Z"

]
```

**Field Descriptions:**
- `status` (required): Generator status (max 20 characters, e.g., "off", "on", "starting", "error")
- `fuel_consumption_lph` (optional, default: 0.0): Fuel consumption in liters per hour
- `charging_power_kw` (optional, default: 0.0): Charging power in kW
- `timestamp` (required): ISO 8601 timestamp in UTC

**Batch Requirements:**
- At least one measurement is required
- Each measurement must have a unique timestamp within the batch
- Duplicate timestamps within a batch will result in `400 Bad Request`

**Response:** `201 Created` with array of created measurement objects

**Error:** If generator feature is disabled, returns `403 Forbidden` with message "Generator feature is not enabled"

### Bulk Measurements Upload

**Endpoint:** `POST /api/installations/{installation_id}/measurements/bulk`

**Description:** Upload measurements for all device types in a single API call. This endpoint accepts optional measurements for Battery, Inverter, Meter, Generator, and EV Charger devices. All measurements are processed atomically in a single transaction - if any device type fails validation, the entire operation is rolled back.

**Request Body:** Object containing optional arrays/objects for each device type
```json
{
  "battery": [
    {
      "soc_percentage": 65.5,
      "power_kw": -2.3,
      "voltage": 48.0,
      "temperature": 25.5,
      "timestamp": "2025-01-11T16:00:00Z"
    }
  ],
  "inverter": {
    "inverter_id": 1,
    "measurements": [
      {
        "power_kw": 3.2,
        "energy_kwh_daily": 12.5,
        "curtailment_percentage": 0.0,
        "timestamp": "2025-01-11T16:00:00Z"
      }
    ]
  },
  "meter": [
    {
      "import_kw": 0.5,
      "export_kw": 1.2,
      "import_kwh": 12.5,
      "export_kwh": 28.8,
      "l1_a": 2.3,
      "l2_a": 2.1,
      "l3_a": 2.4,
      "timestamp": "2025-01-11T16:00:00Z"
    }
  ],
  "generator": [
    {
      "status": "off",
      "fuel_consumption_lph": 0.0,
      "charging_power_kw": 0.0,
      "timestamp": "2025-01-11T16:00:00Z"
    }
  ],
  "ev_charger": {
    "charger_id": 1,
    "measurements": [
      {
        "power_kw": 7.2,
        "energy_kwh": 15.5,
        "source": "battery",
        "revenue_eur": 6.23,
        "timestamp": "2025-01-11T16:00:00Z"
      }
    ]
  }
}
```

**Field Descriptions:**
- `battery` (optional): Array of battery measurements (same format as individual battery endpoint)
- `inverter` (optional): Object with `inverter_id` (integer) and `measurements` (array, same format as individual inverter endpoint)
- `meter` (optional): Array of meter measurements (same format as individual meter endpoint)
- `generator` (optional): Array of generator measurements (same format as individual generator endpoint)
- `ev_charger` (optional): Object with `charger_id` (integer) and `measurements` (array, same format as individual EV charger endpoint)

**Requirements:**
- At least one device type must be provided with at least one measurement
- Each measurement type follows the same validation rules as its individual endpoint
- Duplicate timestamps within a batch for any device type will result in `400 Bad Request`
- All measurements are processed in a single transaction - if any fails, all are rolled back
- Feature flags are checked (generator and EV charger must be enabled if used)
- Device IDs (inverter_id, charger_id) must exist and belong to the installation

**Response:** `201 Created` with grouped response object
```json
{
  "battery": [...],
  "inverter": [...],
  "meter": [...],
  "generator": [...],
  "ev_charger": [...]
}
```

**Error Responses:**
- `400 Bad Request`: No measurements provided, empty arrays, or duplicate timestamps within a batch
- `403 Forbidden`: Installation ID mismatch, or feature not enabled (generator/EV charger)
- `404 Not Found`: Inverter or EV charger not found
- `409 Conflict`: Duplicate measurement timestamps exist in database

**Example:**
```bash
curl -X POST "http://jsenergyapi.aitech.work/api/installations/1/measurements/bulk" \
  -H "X-Device-Token: ${DEVICE_TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "battery": [{"soc_percentage": 65.5, "power_kw": -2.3, "timestamp": "2025-01-11T16:00:00Z"}],
    "meter": [{"import_kw": 0.5, "export_kw": 1.2, "timestamp": "2025-01-11T16:00:00Z"}]
  }'
```

## Code Examples

### Python Example

```python
import os
import requests
from datetime import datetime, timezone, timedelta

# Configuration from environment variables
API_BASE_URL = os.getenv("API_BASE_URL", "http://jsenergyapi.aitech.work")
DEVICE_TOKEN = os.getenv("DEVICE_TOKEN")
INSTALLATION_ID = int(os.getenv("INSTALLATION_ID"))

# Headers for all requests
headers = {
    "X-Device-Token": DEVICE_TOKEN,
    "Content-Type": "application/json"
}

def send_battery_measurement(soc_percentage, power_kw, voltage=None, temperature=None):
    """Send single battery measurement to API."""
    return send_battery_measurements([{
        "soc_percentage": soc_percentage,
        "power_kw": power_kw,
        "voltage": voltage,
        "temperature": temperature,
        "timestamp": datetime.now(timezone.utc).isoformat().replace("+00:00", "Z")
    }])

def send_battery_measurements(measurements):
    """Send batch of battery measurements to API."""
    url = f"{API_BASE_URL}/api/installations/{INSTALLATION_ID}/measurements/battery"
    
    # Ensure timestamps are formatted correctly
    formatted_measurements = []
    for m in measurements:
        if isinstance(m.get("timestamp"), datetime):
            m["timestamp"] = m["timestamp"].isoformat().replace("+00:00", "Z")
        formatted_measurements.append(m)
    
    response = requests.post(url, json=formatted_measurements, headers=headers)
    response.raise_for_status()
    return response.json()

def get_inverters():
    """Get list of inverters for installation."""
    url = f"{API_BASE_URL}/api/installations/{INSTALLATION_ID}/inverters"
    response = requests.get(url, headers=headers)

    return response.json()

def send_bulk_measurements(battery=None, inverter_id=None, inverter_measurements=None, 
                          meter=None, generator=None, charger_id=None, ev_charger_measurements=None):
    """Send bulk measurements for all device types in a single API call."""
    url = f"{API_BASE_URL}/api/installations/{INSTALLATION_ID}/measurements/bulk"
    
    payload = {}
    
    if battery:
        formatted_battery = []
        for m in battery:
            if isinstance(m.get("timestamp"), datetime):
                m["timestamp"] = m["timestamp"].isoformat().replace("+00:00", "Z")
            formatted_battery.append(m)
        payload["battery"] = formatted_battery
    
    if inverter_id and inverter_measurements:
        formatted_inverter = []
        for m in inverter_measurements:
            if isinstance(m.get("timestamp"), datetime):
                m["timestamp"] = m["timestamp"].isoformat().replace("+00:00", "Z")
            formatted_inverter.append(m)
        payload["inverter"] = {
            "inverter_id": inverter_id,
            "measurements": formatted_inverter
        }
    
    if meter:
```

## Error Handling

### HTTP Status Codes

- `201 Created` - Measurements successfully created (returns array of created measurements)
- `400 Bad Request` - Invalid request data (validation error, empty batch, or duplicate timestamps within batch)
- `401 Unauthorized` - Invalid or missing device token
- `403 Forbidden` - Installation ID mismatch, feature disabled, or device inactive
- `404 Not Found` - Installation, inverter, or charger not found
- `409 Conflict` - Duplicate measurement detected (measurement with same timestamp already exists)
- `422 Unprocessable Entity` - Validation error in request body
- `500 Internal Server Error` - Server error

### Error Response Format

All errors follow this format:

```json
{
  "detail": "Error message description"
}
```

### Common Errors

**Invalid Token:**
```json
{
  "detail": "Invalid or inactive device token"
}
```
**Solution:** Verify your device token is correct and the device is active.

**Installation ID Mismatch:**
```json
{
  "detail": "Installation ID mismatch"
}
```
**Solution:** Ensure the `installation_id` in the URL matches your device's installation.

**Inverter/Charger Not Found:**
```json
{
  "detail": "Inverter not found"
}
```
**Solution:** Verify the inverter/charger ID exists and belongs to your installation.

**Feature Disabled:**
```json
{
  "detail": "Generator feature is not enabled"
}
```
**Solution:** The feature is not enabled for your installation. Contact administrator.

**Duplicate Timestamps in Batch:**
```json
{
  "detail": "Duplicate timestamps found in batch: ['2025-01-11T16:00:00Z']. Each measurement must have a unique timestamp."
}
```
**Solution:** Ensure each measurement in a batch has a unique timestamp.

**Duplicate Measurement (409 Conflict):**
```json
{
  "detail": "Duplicate measurement detected. A measurement with the same timestamp already exists for this installation. Original error: ..."
}
```
**Solution:** A measurement with this timestamp already exists. Use a different timestamp or check if you've already submitted this measurement.

### Retry Logic

Implement retry logic with exponential backoff for network errors:

```python
import time
import requests
from requests.exceptions import RequestException

def send_with_retry(url, data, headers, max_retries=3):
    """Send request with exponential backoff retry."""
    for attempt in range(max_retries):
        try:
            response = requests.post(url, json=data, headers=headers, timeout=10)
            response.raise_for_status()
            return response.json()
        except RequestException as e:
            if attempt == max_retries - 1:
                raise
            wait_time = 2 ** attempt  # Exponential backoff: 1s, 2s, 4s
            time.sleep(wait_time)
            print(f"Retry {attempt + 1}/{max_retries} after {wait_time}s...")
```

## Best Practices

### 1. Timestamp Format

Always use ISO 8601 format with UTC timezone:
- ✅ `"2025-01-11T16:00:00Z"`
- ❌ `"2025-01-11 16:00:00"`
- ❌ `"2025-01-11T16:00:00+00:00"` (works but prefer Z)

### 2. Data Frequency and Batching

- **Battery, Inverter, Meter**: Send every 3-5 seconds for real-time monitoring
- **Generator**: Send every 1 minute or on state change
- **EV Chargers**: Send every 1 minute or on state change

**Batch Operations:**
- All measurement endpoints support batch operations (arrays of measurements)
- Batch multiple measurements together to reduce API calls
- Each measurement in a batch must have a unique timestamp
- Ideal for buffering measurements during network outages and sending them in bulk
- Single measurements can still be sent as arrays with one element

**Bulk Upload:**
- Use the `/measurements/bulk` endpoint to send measurements for all device types in a single API call
- Reduces network overhead and improves efficiency when collecting data from multiple devices simultaneously
- All measurements are processed atomically - if any device type fails, the entire operation is rolled back
- Ideal for edge devices that collect data from multiple sources at the same time

### 3. Error Handling

- Always check HTTP status codes
- Implement retry logic for network failures
- Log errors for debugging
- Buffer measurements locally during network outages

### 4. Security

- Never log or expose device tokens
- Use HTTPS in production
- Store tokens securely (environment variables, not in code)
- Rotate tokens if compromised

### 5. Data Validation

- Validate data before sending (e.g., SOC percentage 0-100)
- Handle missing optional fields gracefully
- Ensure timestamps are in UTC

### 6. Network Resilience

- Implement automatic reconnection
- Buffer measurements during outages and send in batches when connection is restored
- Use batch operations to reduce API calls and improve efficiency
- Use appropriate timeouts (recommend 10-30 seconds)
- Handle rate limiting (1000 requests/minute per device)
- Batch operations help stay within rate limits by reducing the number of requests

### 7. Monitoring

- Monitor your device's `last_seen_at` timestamp in the dashboard
- Set up alerts for authentication failures
- Track measurement submission success rates

## Summary

1. **Device Registration**: Done by administrator via dashboard/API
2. **Token Storage**: Save token securely from registration response
3. **Authentication**: Include token in `X-Device-Token` header
4. **Data Submission**: POST measurements to appropriate endpoints (arrays for batch support)
5. **Batch Operations**: All endpoints support batch operations - send arrays of measurements to reduce API calls
6. **Bulk Upload**: Use `/measurements/bulk` endpoint to send all device types in a single call for maximum efficiency
7. **Error Handling**: Implement retry logic and proper error handling
8. **Best Practices**: Follow security, validation, and resilience guidelines
